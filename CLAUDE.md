# Trad Tune Book — Project Notes for Claude

Personal offline-first PWA for tracking Irish traditional music tunes (tunes I know vs. tunes I'm learning), practicing with a random draw, and sharing with a small group of friends. Lives at https://tradtunebook.com.

## Architecture

- **Frontend**: React 18 + Vite, PWA via vite-plugin-pwa / Workbox
- **Local storage**: Dexie.js (IndexedDB), offline-first
- **Sync / auth**: Dexie Cloud (https://z3e2bhpjn.dexie.cloud) — email OTP login, free eval tier
- **Hosting**: Cloudflare Pages, auto-deploy from `main` branch on GitHub push
- **Domain**: tradtunebook.com (DNS at Cloudflare, registered at GoDaddy)
- **Design**: warm paper palette (#1a1814 ink, #f4ede0 paper, #8b3a1f accent, #2e5339 green), Fraunces display + IBM Plex Sans UI

## Data model

Three user tables: `tunes`, `profiles`, `config`. Plus Dexie Cloud's built-in `realms`, `members`, `roles`.

Tune fields: `name`, `type` (reel/jig/slip_jig/hornpipe/polka/slide/mazurka/waltz/march/air/barndance), `musicalKey`, `status` (know/learning), `notes`, `recordingUrl`, `dateAdded`, `lastPracticed`, `sessionOrgId`, `sessionOrgMatchStatus` (pending/matched/ambiguous/unmatched), `sessionOrgCandidates`, `sessionOrgManual` (boolean — protects manual overrides from auto-match).

## Sharing model

All members of the shared realm (`rlm-shared-*`) see everyone's tunes. The default list view is "My tunes" with a toggle to "All tunes". Peer-suggestion appears when adding a tune name someone else already has. Copy-to-mine action for another user's tune (starts as "learning").

Realm is created at runtime by the first user; the ID is stored in the `config` table under key `sharedRealmId`. Invites go through `db.members.add({...invite: true...})` — the **field name is `invited`, NOT `invitedDate`** (getting this wrong causes silent server reverts).

## thesession.org integration

Two-tier matching: (1) bundled `public/data/aliases.json` fast path (run `npm run fetch-aliases` to refresh from the adactio/TheSession-data repo), (2) live API `https://thesession.org/tunes/search?q=X&format=json` fallback. Background queue in `src/session-org/queue.js` drains pending matches every 5 min when online. Skips tunes with `sessionOrgManual: true` so user overrides stick.

## Local dev

    npm install
    npm run fetch-aliases   # one-time; regenerates aliases.json (~10 MB)
    npm run dev             # http://localhost:5173

## Deploy

Auto: `git push origin main` → Cloudflare Pages rebuilds in ~2–3 min. Build command: `npm run fetch-aliases && npm run build`, output `dist`, env var `VITE_DEXIE_CLOUD_URL`.

Manual overrides (whitelist, etc.): use the Dexie Cloud CLI locally — `npx dexie-cloud whitelist <origin>` (note: NO `add` keyword). List with `npx dexie-cloud whitelist` (no args).

## Gotchas

- Dexie Cloud schema strings must be single-line comma-separated — whitespace/newlines break `createObjectStore`.
- `db.cloud.currentUser` and `db.cloud.invites` are Observables, not Promises — use `useObservable`, not `useLiveQuery`.
- `useLiveQuery(() => cond ? obs : [])` returning an Observable object breaks `.filter()` etc — use `useObservable(db.cloud?.invites) ?? []` instead.
- `tuneOps.update` spreads the entire form back to the DB, so any field changed outside the form (via direct `db.tunes.update`) also needs a matching `setForm` call in the editor, or the save will revert it.
- Dexie Cloud Member schema field is `invited` (not `invitedDate`). Permissions `update` shape is `{[table]: string[] | "*"}`, not a flat array.
- `VITE_DEXIE_CLOUD_URL` in `.env` must be the `dbUrl` (e.g. `https://z3e2bhpjn.dexie.cloud`), NOT `serviceUrl` (just `https://dexie.cloud`).
